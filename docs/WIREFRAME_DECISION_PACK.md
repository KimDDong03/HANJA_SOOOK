# WIREFRAME_DECISION_PACK

## 목적

현재 레포 기준으로 한자쏘옥 화면 구조를 다시 정리하기 위한 의사결정 자료다.
이 문서는 기능 인벤토리, IA 후보, 사이트맵 후보, 탭/내비 구조 비교를 하나로 합친다.
아직 와이어프레임 이미지는 만들지 않는다.

## 현재 논의 반영 추천 방향

기존 후보 C의 하단 탭 앱형을 유지하되, 탭 이름과 역할을 한자쏘옥 장기 구조에 맞게 조정한다.

```txt
Splash
→ Role Select
  → 학생
    → Student Setup / Login
    → App Shell
      → 홈
      → 학습
      → 도전
      → 설정
  → 학부모
    → Student Links / 보호자 연결 및 진행 확인
  → 선생님
    → Teacher Preview / 교사용 데모
```

권장 탭:

| 탭 | 역할 | 포함 화면/기능 |
|---|---|---|
| 홈 | 오늘 할 일을 바로 시작하는 미션 허브 | 오늘의 한자, 오늘의 미션, XP/성장 요약, 이어서 학습 |
| 학습 | 한자 이해와 쓰기 훈련 | 한자 카드, 획순 보기, 단계별 쓰기 연습, 자유쓰기 채점 |
| 도전 | 퀴즈/게임/경쟁 | 한자-훈음 퀴즈, 훈음-한자 퀴즈, 뜻-한자 선택, 판뒤집기, 반 랭킹 |
| 설정 | 역할/연결/데모성 기능 격리 | 학생 연결, 반 참여, 교사 미리보기, 계정/앱 정보 |

Growth는 별도 하단 탭으로 고정하지 않고 `홈 요약`, `결과 화면`, `성장 상세 화면`으로 분산한다.
이렇게 하면 학생의 기본 탭은 단순해지고, 경쟁/게임 기능은 `도전`으로 명확히 분리된다.

## 진입/역할 선택 방향

역할 선택 화면은 학생 홈이 아니라 첫 진입 또는 데모 진입 화면으로 둔다.

```txt
학생: 한자를 재미있게 배워요
학부모: 우리 아이 학습 현황을 확인해요
선생님: 학생들의 학습을 관리해요
```

권장 동작:

- 첫 실행 또는 프로필이 없을 때 역할 선택을 보여준다.
- 학생 선택 후 이름/소속/학년 선택으로 이동한다.
- 학생 프로필이 있으면 다음 실행부터 바로 학생 앱 홈으로 진입한다.
- 역할 변경, 학부모 연결, 선생님 미리보기는 설정 탭에서 접근한다.

## 선택안 v1

현재 논의 기준으로 아래 구조를 우선 선택안으로 둔다.
단, 생성형 와이어프레임의 시각 스타일은 참고용이며 실제 구현 전 텍스트와 세부 상태는 별도로 정리한다.

| 화면 | 선택안 | 결정 메모 |
|---|---|---|
| Role Select | 학생 우선형 | 첫 진입에서 학생을 기본 역할로 크게 보여주고, 학부모/선생님은 보조 진입으로 둔다. |
| Student Setup | 학교 검색 결합형 | 기존 MVP의 학교 검색, 이름, 학년 선택 정책과 맞춘다. |
| Home | 오늘 미션 중심 | 오늘의 한자와 오늘 할 일을 가장 크게 보여준다. |
| Learn | 단계형 학습 루트 | 한자 카드부터 자유쓰기 채점까지 학습 순서를 명확히 보여준다. |
| Hanja Card | 레슨 브리핑형 | 정보 확인 후 획순/쓰기 단계로 자연스럽게 이동한다. |

| Guided Writing | 반복 훈련형 | 여러 번 따라 쓰며 가이드를 점점 줄이는 구조로 간다. |

| Challenge | 반 경쟁 중심 | 도전 탭은 퀴즈/게임보다 반 경쟁 동기를 먼저 보여준다. |
| Quiz Modes | 모드 카드형 | 퀴즈 종류를 명확히 나누고 사용자가 선택한다. |
| Quiz Play | 1, 2, 4번 채택 | `한자→훈음`, `훈음→한자`, `혼합 퀴즈`은 서로 다른 퀴즈 모드로 채택한다. 3번 `뜻→한자 속도형`은 별도 게임/도전 모드로 분리한다. |
| Flip Board Game | 후보 모두 채택 | 기본형, 리버스형, 반 경쟁형, 라운드 전략형은 대안이 아니라 게임 모드/상태로 모두 사용한다. |
| Class Ranking | 통합 리그형 | XP, 오늘 도전, 판뒤집기 기록을 필터/탭으로 전환한다. |
| Result | 성장 보상형 | 결과는 점수보다 XP, 레벨, 다음 행동을 중심에 둔다. |
| Settings | 역할별 설정형 | 첫 시작 Role Select 이후 선택된 role에 따라 설정 화면 내용을 다르게 보여준다. |

## 결정해야 할 핵심 질문

1. 학생 홈을 학습 허브로 유지할지, 성장/미션 중심 대시보드로 바꿀지
2. 한자 카드, 쓰기, 퀴즈, 게임을 하나의 학습 흐름으로 묶을지, 홈에서 각각 독립 진입시키는 구조로 둘지
3. Growth를 상시 하단 탭으로 둘지, 홈의 보상/결과 흐름에서만 접근하게 둘지
4. Teacher Preview와 Student Links를 MVP 화면 구조 안에 노출할지, 데모/설정성 보조 화면으로 격리할지
5. 초등학생 대상 앱으로서 탐색 자유도를 높일지, 오늘의 미션 중심으로 다음 행동을 강하게 안내할지

## 학습 모드 구조

학습 탭은 `한자를 익히는 과정`을 담당한다.
쓰기 난이도는 안내를 점점 줄이는 방식으로 구성한다.

```txt
한자 카드
→ 획순 보기
→ 획순 보고 따라쓰기
→ 희미한 가이드 보고 쓰기
→ 가이드 없이 기억해서 쓰기
→ 완전 자유쓰기 채점
→ 결과
```

권장 모드:

| 모드 | 안내 수준 | 목적 | 결과 |
|---|---|---|---|
| 획순 보고 따라쓰기 | 높음 | 획순/방향 학습 | 완료/기본 점수 |
| 가이드 쓰기 | 중간 | 글자 형태 안정화 | 정확도/별 |
| 기억 쓰기 | 낮음 | 기억 기반 재현 | 정확도/별/XP |
| 자유쓰기 채점 | 없음 | 실력 확인 | 점수/랭크/재도전 |
| Hanja Demo 채점 | 없음 | 아무 안내 없이 쓰고 채점 | 별도 데모/실험 진입 가능 |

원칙:

- 획순 데이터가 있으면 guided mode부터 시작한다.
- 획순 데이터가 없으면 자유쓰기 채점으로 fallback한다.
- missing data 때문에 학습 흐름을 막지 않는다.
- 점수/XP 계산은 UI가 아니라 도메인 서비스에서 처리한다.

## 도전 모드 구조

도전 탭은 `맞히기`, `속도`, `경쟁`을 담당한다.
학습 탭이 개인 훈련이라면, 도전 탭은 반복 플레이와 반 단위 동기 부여를 담당한다.

### 기본 도전

| 모드 | 문제 입력 | 답변 방식 | 핵심 피드백 |
|---|---|---|---|
| 한자 보고 훈음 맞추기 | 한자 | 음/뜻 선택 또는 입력 | 정답/오답, 해설 |
| 훈음 보고 한자 맞추기 | 음/뜻 | 한자 선택 | 정답/오답, 예문 |
| 뜻 보고 한자 선택 | 뜻 | 4~6개 한자 선택 | 시간, 콤보, 점수 |
| 오늘의 도전 10문항 | 혼합 | 선택형 중심 | 점수, XP, 반 랭킹 반영 |

Quiz Play 원칙:

- `한자 보고 훈음 맞추기`, `훈음 보고 한자 맞추기`, `혼합 퀴즈`은 화면 후보가 아니라 채택할 퀴즈 모드다.
- 각 모드는 같은 퀴즈 플레이 구조를 공유하되, 문제 카드와 답변 영역만 모드에 맞게 바뀐다.
- `뜻 보고 한자 선택`은 속도/콤보가 핵심이므로 일반 Quiz Play보다 별도 게임성 도전으로 분리한다.

### 판뒤집기 게임

한컴타자 판뒤집기에서 착안한 반 경쟁형 게임으로 기획한다.
정확한 외부 규칙 복제가 아니라, `판을 많이 뒤집어 점유한다`는 구조만 한자 학습에 맞게 적용한다.
판뒤집기는 앱이 문제를 하나씩 순서대로 제시하는 방식이 아니다.
학생이 직접 뒤집고 싶은 판을 고르고, 그 판에 적힌 한자/훈음/뜻을 보고 답을 작성한다.

기본형:

```txt
판 여러 개에 훈음/뜻 표시
→ 학생이 풀고 싶은 판을 선택
→ 학생이 해당 한자를 쓴다
→ 채점 통과 시 내 색으로 판 뒤집기
→ 제한시간 동안 더 많은 판을 점유한 학생/팀 승리
```

리버스형:

```txt
판 여러 개에 한자 표시
→ 학생이 풀고 싶은 판을 선택
→ 학생이 훈음 또는 뜻을 작성한다
→ 정답이면 내 색으로 판 뒤집기
→ 제한시간 동안 더 많은 판을 점유한 학생/팀 승리
```

채택할 판뒤집기 모드:

| 모드 | 판에 표시되는 내용 | 답변 방식 | 목적 |
|---|---|---|---|
| 기본형 | 훈음/뜻 | 한자 쓰기 | 쓰기 기반 점유 경쟁 |
| 리버스형 | 한자 | 훈음 또는 뜻 작성 | 이해/기억 기반 점유 경쟁 |
| 반 경쟁형 | 기본형 또는 리버스형 | 모드별 답변 | 반 친구들과 점수/점유 수 경쟁 |
| 라운드 전략형 | 여러 난이도 판 | 선택한 판에 답변 | 쉬운 판을 빠르게 뒤집을지 어려운 판으로 고득점할지 선택 |

와이어프레임에 필요한 상태:

- 내 점유 판 수
- 상대/반 평균 점유 판 수
- 남은 시간
- 선택한 판의 내용
- 판별 점유 상태
- 입력 영역: 쓰기 캔버스 또는 텍스트/선택 입력
- 정답/오답 즉시 피드백
- 최종 순위와 XP 지급

MVP 절충:

- 1차는 실시간 대전이 아니라 `비동기 반 랭킹`으로 시작한다.
- 같은 반 학생들의 누적 XP, 오늘 도전 점수, 판뒤집기 최고 점수를 보여준다.
- 실시간 점유전은 이후 확장으로 둔다.

## IA 후보

### 후보 A. 오늘의 미션 중심형

```txt
Splash
→ Login
→ Home
  → 오늘의 한자 시작
    → Hanja Card
    → Writing
    → Quiz
    → Result
    → 다음 한자 또는 Home
  → Game
  → Growth
  → 보조 메뉴: Student Links / Teacher Preview
```

특징:

- 학생에게 다음 행동이 가장 명확하다.
- MVP 성공 기준의 학습 루프를 가장 직접적으로 보여준다.
- 한자 카드, 쓰기, 퀴즈가 하나의 시퀀스로 느껴진다.

장점:

- 초등학생 대상에서 인지 부하가 낮다.
- 데모 진행자가 설명하기 쉽다.
- `오늘의 학습 N/4`와 XP 보상 구조가 자연스럽다.

단점:

- 퀴즈/게임을 자유롭게 반복하고 싶은 구조에는 약하다.
- Teacher Preview, Student Links 같은 보조 기능을 어디에 둘지 별도 규칙이 필요하다.

선택하면 좋은 경우:

- 1차 MVP를 "학습 흐름이 끊기지 않는 시연 앱"으로 강하게 가져갈 때
- 홈을 대시보드보다 미션 시작 화면에 가깝게 만들고 싶을 때

### 후보 B. 활동 허브형

```txt
Splash
→ Login
→ Home
  → Hanja Card
  → Writing
  → Quiz
  → Game
  → Growth
  → Teacher Preview
  → Student Links
```

특징:

- 현재 홈 구현에 가장 가깝다.
- 모든 주요 화면으로 직접 진입할 수 있다.
- 데모용으로 화면 접근성이 좋다.

장점:

- 구현 변경 폭이 작다.
- 기능이 많아 보이고, 시연 중 특정 화면으로 이동하기 쉽다.
- 각 기능의 독립 완성도를 보여주기 좋다.

단점:

- 학생 입장에서는 무엇을 먼저 해야 하는지 약해질 수 있다.
- 학습 루프보다 기능 목록처럼 보일 위험이 있다.
- Teacher Preview와 Student Links가 학생용 활동과 같은 위계로 보이면 MVP 메시지가 흐려진다.

선택하면 좋은 경우:

- 빠른 데모 접근성과 구현 안정성을 우선할 때
- 아직 학습 시퀀스의 세부 순서를 확정하지 않았을 때

장점:

- Home, Learn, Growth, More의 역할이 분명하다.
- Teacher Preview와 Student Links를 More로 격리하기 쉽다.

단점:

- MVP 기준으로는 구조가 다소 무거울 수 있다.

- App Shell, 탭 상태, 내부 라우팅 정리가 필요하다.

선택하면 좋은 경우:

## 화면별 구조 결정 메모

### Splash

- 유지: 1초 후 Login 이동
- 와이어프레임 결정: 로고 중심 단순 화면이면 충분하다.
- 보류: 세션 복원 후 Home 이동은 이후 단계.

### Login

- 필수 요소: 학교 검색, 결과 목록, 선택 학교, 이름, 학년, 시작하기
- 후보:
  - 단일 화면형: 빠르고 간단하지만 입력 요소가 많다.
  - 단계형: 학교 선택 → 학생 정보 입력으로 명확하지만 화면 수가 늘어난다.
- 추천 판단: 모바일 첫 진입에서는 단계형이 더 안정적이나, MVP 구현 범위는 단일 화면형이 작다.

### Home

- 반드시 보여야 할 것: 인사, 오늘의 한자, 진행 N/4, 주요 시작 CTA, Growth 접근
- 결정 필요:
  - 메인 CTA를 `오늘의 한자 시작` 하나로 강하게 둘지
  - 퀴즈/게임/쓰기 카드를 같은 위계로 둘지
  - Teacher Preview와 Student Links를 홈에 직접 노출할지
- 추천 판단: 상단은 오늘의 미션, 중단은 학습 활동, 하단은 보조 메뉴로 위계를 나눈다.

### Hanja Card

- 역할 후보:
  - 정보 카드: 뜻/음/예문/이미지 학습
  - 레슨 시작 브리핑: 쓰기/퀴즈 전 핵심 정보 확인
- 추천 판단: MVP에서는 브리핑 역할을 강화하고 `따라쓰기`를 가장 강한 다음 CTA로 둔다.

### Writing

- 필수: 가이드 글자, 캔버스, 다시쓰기, 완료, 정확도/별
- 결정 필요:
  - 획순 데이터가 없을 때 자유쓰기 안내를 어느 정도 노출할지
  - 정확도를 실시간으로 보여줄지, 완료 후 보여줄지
- 추천 판단: 초등학생에게는 실시간 수치보다 완료 후 별/응원 피드백이 더 적합하다.

### Quiz

- 필수: N/10, 문제, 4지선다, 즉시 피드백, 다음
- 결정 필요:
  - 퀴즈를 오늘의 한자 흐름 끝에 붙일지
  - 홈에서 바로 시작하는 독립 모드도 유지할지
- 추천 판단: 둘 다 가능하되 와이어프레임에서는 "오늘의 한자 복습 퀴즈"라는 맥락을 명확히 한다.

### Game

- 필수: 뜻 프롬프트, 한자 선택지 4~6개, 시간 게이지, 점수, 콤보
- 결정 필요:
  - 학습 필수 흐름인지 보너스 활동인지
- 추천 판단: MVP에서는 보너스 활동으로 두는 편이 학습 루프를 단순하게 유지한다.

### Result

- 필수: 메시지, 별점, 점수, 정답률/정확도, XP/코인, 다시 도전, 다음/홈
- 결정 필요:
  - 쓰기/퀴즈/게임 결과를 한 템플릿으로 통합할지
- 추천 판단: 공통 결과 템플릿을 쓰고 활동별 지표 라벨만 다르게 한다.

### Growth

- 필수: 캐릭터, 레벨, XP bar, 배지, 능력치 바
- 결정 필요:
  - 홈의 일부인지 별도 목적지인지
- 추천 판단: 별도 화면은 유지하되 홈 상단/결과 화면에서 일부 요약을 보여준다.

### Teacher Preview

- 필수: 샘플 학급명, 학생 수, 과제 요약, 학생별 표
- 결정 필요:
  - 학생 앱에서 얼마나 노출할지
- 추천 판단: `More` 또는 데모 메뉴로 격리한다. 학생 홈의 주요 학습 카드와 같은 위계는 피한다.

### Student Links

- 현재 구현 존재: 학생 연결 코드, 보호자 연결, 반 참여, 참여한 반, 연결된 학생
- 스펙상 MVP 필수는 아님.
- 결정 필요:
  - MVP 데모에 포함할지
  - 포함한다면 보호자/교사 연결 설정으로 볼지
- 추천 판단: 구조안 선택 전에는 주요 IA가 아니라 보조 기능으로 분류한다.

### Settings

- 설정 탭은 모든 role에 같은 목록을 보여주는 화면이 아니다.
- 첫 시작의 Role Select에서 선택한 role에 따라 설정 항목을 다르게 보여준다.
- 학생 role: 내 프로필, 학년/학교 정보, 반 참여, 역할 변경, 앱 정보
- 학부모 role: 학생 연결, 연결된 학생, 학습 현황 진입, 역할 변경, 앱 정보
- 선생님 role: 교사 미리보기, 반/학생 샘플 관리, 역할 변경, 앱 정보
- 공통: 역할 변경은 설정에서 가능하되, 학생 학습 흐름의 주요 CTA와 같은 위계로 노출하지 않는다.

## 추천 의사결정안

현재 우선안은 상단의 `선택안 v1`이다.
기존 A/B/C/D 후보는 비교 검토용으로 남겨두되, 앞으로의 와이어프레임과 구현 재정리는 아래 방향을 기준으로 한다.

```txt
Role Select
→ Student Setup
→ App Shell
  → 홈
  → 학습
  → 도전
  → 설정
```

핵심 판단:

- 홈은 오늘 미션 중심으로 둔다.
- 학습은 단계형 쓰기 루틴으로 둔다.
- 도전은 퀴즈, 판뒤집기, 반 랭킹을 포함하는 경쟁 허브로 둔다.
- 설정은 Role Select에서 선택한 role에 따라 다른 항목을 보여준다.

## 선택 전 체크리스트

- 학생 홈에서 첫 번째 CTA는 무엇인가?
- 쓰기 완료 후 다음 화면은 Result인가, Quiz인가?
- Quiz와 Game은 오늘의 한자 흐름에 포함되는가, 독립 활동인가?
- Growth는 하단 탭인가, 홈/결과에서 진입하는 보상 화면인가?
- Student Links는 이번 와이어프레임에 포함하는가?
- Teacher Preview는 학생 앱 IA 안에 보이는가, 데모 메뉴로 숨기는가?
- `student-links` 라우트를 공식 MVP 범위로 승격할 것인가, 보조/실험 기능으로 둘 것인가?

## 와이어프레임 제작 전 필요한 선택

아래 중 하나를 고르면 다음 단계에서 실제 화면 와이어프레임을 만들 수 있다.

| 선택지 | 구조 | 적합한 다음 작업 |
|---|---|---|
| A | 오늘의 미션 중심형 | Home, Hanja Card, Writing, Result를 먼저 와이어프레임 |
| B | 활동 허브형 | 현재 홈의 카드 위계와 보조 기능 분리안 와이어프레임 |
| C | 하단 탭 앱형 | App Shell, 탭별 첫 화면, More 구조 와이어프레임 |
| D | 레슨 플레이어형 | Lesson Player 플로우와 단계별 CTA 와이어프레임 |
